Guida introduttiva

Git, spiegato davvero da zero.

Git è il sistema con cui il mondo del software tiene traccia di ogni modifica al codice. Sembra ostico all'inizio, ma poggia su pochi concetti solidi. Questa guida ti porta dal primo repository fino a GitHub, branch e tag, un comando alla volta.

01

Cos'è Git, e perché

Git è un sistema di controllo di versione: un programma che fotografa il tuo progetto nel tempo, così puoi vedere com'era ieri, tornare indietro quando rompi qualcosa, e lavorare in molti sullo stesso codice senza pestarvi i piedi. È stato creato nel 2005 da Linus Torvalds per sviluppare il kernel Linux, ed è oggi lo standard di fatto.

Due idee lo rendono diverso dai suoi predecessori:

  • È distribuito. Non esiste un solo archivio centrale di cui fidarsi: quando cloni un progetto, scarichi l'intera cronologia sul tuo computer. Puoi lavorare, salvare versioni e consultare la storia anche offline.
  • Ragiona per istantanee, non per differenze. Ad ogni salvataggio (un commit) Git memorizza una fotografia dello stato dei file, non solo le righe cambiate. Questo rende operazioni come cambiare ramo o tornare indietro estremamente rapide.
Git non è GitHub

Git è il programma che gira sul tuo computer. GitHub (come GitLab o Bitbucket) è un servizio online che ospita repository Git e aggiunge collaborazione, issue e pull request. Useremo Git in locale e GitHub come remoto.

02

Installazione

Su molti sistemi Git è già presente. Verifica con:

git --version

Se manca, installalo con il gestore di pacchetti del tuo sistema.

macOS

# con Homebrew (consigliato)
brew install git

# in alternativa, gli strumenti da riga di comando di Xcode
xcode-select --install

Linux

# Debian / Ubuntu
sudo apt update && sudo apt install git

# Fedora
sudo dnf install git

# Arch Linux
sudo pacman -S git
Pagina di download ufficiale
E Windows?

Su Windows la cosa migliore è usare Git dentro WSL (Windows Subsystem for Linux): in questo modo ricadi nel caso Linux e segui i comandi della distribuzione che hai scelto.

03

Come ragiona Git: le tre aree

Questo è il concetto che sblocca tutto il resto. In un progetto gestito da Git, i tuoi file vivono in tre stati, e ogni comando li fa passare dall'uno all'altro.

AreaCos'è
Working Directory
directory di lavoro		La cartella del progetto come la vedi nel file manager o nel terminale. Qui modifichi davvero i file.
Staging Area
o index		Un'anticamera dove prepari il prossimo commit. Decidi quali modifiche faranno parte della prossima fotografia.
Repository
la cartella .git		L'archivio permanente: qui Git registra ogni commit con la cronologia completa.

Durante la vita di un progetto, ogni file attraversa stati ben precisi. All'inizio Git non lo conosce (untracked); dopo il primo add lo segue e lo considera invariato (unmodified); appena lo modifichi diventa modified; quando lo indicizzi è staged, pronto per il commit.

04

Configurazione: identità e opzioni

Prima del primo commit, Git deve sapere chi sei: ogni commit viene firmato con un nome e una email. Attenzione, questa è la tua identità d'autore, non una password: serve a etichettare i commit, non ad autenticarti. L'autenticazione verso GitHub la gestiremo più avanti con le chiavi SSH.

Globale o locale?

La configurazione vive su tre livelli, dal più generale al più specifico. Il livello più specifico vince sempre.

LivelloOpzioneFileVale per
System--system/etc/gitconfigtutti gli utenti della macchina
Global--global~/.gitconfigtutti i tuoi repository
Local--local.git/configsolo questo repository

Imposta l'identità a livello globale una volta sola, sul tuo computer:

git config --global user.name "Mario Rossi"
git config --global user.email "mario.rossi@example.com"

Hai un'email diversa per un singolo progetto (ad esempio quella di lavoro)? Sovrascrivi il valore solo lì, dentro la cartella del repository, senza --global:

# eseguito dentro la cartella del progetto
git config user.email "mario.rossi@azienda.com"
E la password?

Git non memorizza una password per firmare i commit: nome ed email bastano. Le credenziali servono solo per comunicare con un remoto come GitHub, e il modo più comodo e sicuro è la chiave SSH che vedrai nel capitolo 07.

Altre opzioni utili da subito

# chiama "main" il ramo iniziale dei nuovi repository
git config --global init.defaultBranch main

# l'editor usato per i messaggi di commit
git config --global core.editor "nano"        # oppure "vim", "emacs", "code --wait"

# colori leggibili nell'output
git config --global color.ui auto

Per rivedere cosa hai impostato e da quale file proviene ogni valore:

git config --list --show-origin
git config user.email          # mostra un singolo valore

05

Creare un repository locale

Trasformare una cartella qualsiasi in un repository Git richiede un solo comando. git init crea una sottocartella nascosta .git, ed è lì che Git custodirà l'intera cronologia.

# crea la cartella ed entraci
mkdir mio-progetto
cd mio-progetto

# inizializza il repository
git init

# la "prova": ora esiste la cartella .git
ls -a

Da questo momento il progetto è tracciato. Creiamo un file e seguiamone il percorso fino al primo commit:

# 1. crea un file nella working directory
echo "# Il mio progetto" > README.md

# 2. Git lo vede come "untracked"
git status

# 3. spostalo nella staging area
git add README.md

# 4. congelalo nella cronologia con un messaggio
git commit -m "Primo commit: aggiungo il README"
Scrivi buoni messaggi di commit

Usa l'imperativo presente e sii sintetico: "Aggiungo la validazione del form", non "aggiunte varie". Il tuo te stesso del futuro ti ringrazierà leggendo il git log.

Tenere fuori i file inutili: .gitignore

Non tutto va versionato: dipendenze, file temporanei, segreti. Un file .gitignore nella radice elenca i pattern da ignorare.

# dipendenze
node_modules/
.venv/

# sistema operativo
.DS_Store

# backup di emacs
*~
\#*\#
.\#*

# segreti e ambiente
.env
*.log

06

Le azioni di tutti i giorni

Il 90% del lavoro quotidiano con Git ruota attorno a una manciata di comandi. Eccoli, nell'ordine in cui li userai.

git status & git diff — capire dove sei

git status                 # quali file sono modificati o in staging
git diff                   # cosa è cambiato, ma non ancora indicizzato
git diff --staged          # cosa finirà nel prossimo commit

git add — preparare il commit

git add file.txt           # un file specifico
git add cartella/          # un'intera cartella
git add .                  # tutto ciò che è cambiato qui sotto
git add -p                 # scegli interattivamente blocco per blocco

git commit — congelare l'istantanea

git commit -m "Messaggio breve e chiaro"

# scorciatoia: indicizza i file GIÀ tracciati e fa il commit
git commit -am "Sistemo il calcolo del totale"

# commit diretto di un file già tracciato, senza passare da git add
git commit -m "Correggo un refuso" file.txt

Ogni commit punta al precedente (il suo parent), formando la catena che è la storia del progetto. Ognuno è identificato da un hash univoco.

git log — leggere la storia

git log                              # cronologia completa
git log --oneline                    # una riga per commit
git log --oneline --graph --all      # con il grafo dei rami
git show 98ca9                       # i dettagli di un commit

Annullare e correggere

git restore file.txt           # scarta le modifiche non indicizzate
git restore --staged file.txt  # toglie dallo staging (resta modificato)
git commit --amend             # corregge l'ultimo commit (messaggio o file)
Riscrivere la storia ha un costo

Comandi come --amend o git reset cambiano la cronologia. Falli pure in locale, ma evita di riscrivere commit che hai già condiviso con altri tramite push: creeresti confusione per chi ha già scaricato quei commit.

07

GitHub e le chiavi SSH

Per collaborare e fare un backup online colleghi il tuo repository a un remoto. Useremo GitHub, autenticandoci con una chiave SSH: la generi una volta, la registri su GitHub e non dovrai più digitare credenziali.

Il principio è una coppia di chiavi: una privata, che resta gelosamente sul tuo computer, e una pubblica, che pubblichi su GitHub. Chi possiede la privata può autenticarsi; la pubblica da sola non serve a nessuno.

1 · Genera una chiave ed25519

L'algoritmo ed25519 è moderno, veloce e sicuro: è la scelta consigliata.

ssh-keygen -t ed25519 -C "mario.rossi@example.com"

Premi Invio per accettare il percorso predefinito (~/.ssh/id_ed25519) e imposta una passphrase: è la password che protegge la chiave privata se qualcuno mettesse le mani sul file.

2 · Aggiungi la chiave all'agent

L'ssh-agent tiene in memoria la chiave sbloccata, così non reinserisci la passphrase ogni volta.

# avvia l'agent
eval "$(ssh-agent -s)"

# memorizza la chiave nel Portachiavi di macOS
ssh-add --apple-use-keychain ~/.ssh/id_ed25519

Sempre su macOS, conviene rendere la cosa permanente creando (o aggiornando) il file ~/.ssh/config:

Host github.com
  AddKeysToAgent yes
  UseKeychain yes
  IdentityFile ~/.ssh/id_ed25519
# avvia l'agent e aggiungi la chiave
eval "$(ssh-agent -s)"
ssh-add ~/.ssh/id_ed25519

3 · Pubblica la chiave su GitHub

Copia il contenuto della chiave pubblica (il file con estensione .pub):

# macOS: copia direttamente negli appunti
pbcopy < ~/.ssh/id_ed25519.pub

# Linux (o ovunque): stampala e copiala a mano
cat ~/.ssh/id_ed25519.pub

Poi su GitHub: Settings → SSH and GPG keys → New SSH key, incolla il contenuto, dai un nome riconoscibile (es. "MacBook lavoro") e salva.

4 · Verifica la connessione

ssh -T git@github.com

Se vedi un messaggio del tipo "Hi mario! You've successfully authenticated...", è fatta.

HTTPS o SSH?

GitHub offre anche l'accesso via HTTPS con un token. SSH è più comodo a regime: configurato una volta, sparisce dalla tua vista. Riconosci un indirizzo SSH perché inizia con git@github.com:, mentre quello HTTPS con https://.

5 · Collega e sincronizza

Se il progetto esiste già su GitHub, clonalo (scarichi tutto):

git clone git@github.com:utente/repo.git

Se invece parti da un repository locale già creato (capitolo 05), collegalo al remoto e invia i commit:

# collega il remoto, chiamandolo "origin" per convenzione
git remote add origin git@github.com:utente/repo.git
git remote -v                       # verifica i remoti configurati

# rinomina il ramo locale in "main" e invia tutto
git branch -M main
git push -u origin main             # -u lega il ramo locale al remoto

Da qui in poi il dialogo con il remoto si riduce a tre verbi:

ComandoCosa fa
git pushinvia i tuoi commit locali al remoto
git pullscarica i commit dal remoto e li unisce nel tuo ramo
git fetchscarica le novità dal remoto senza toccare il tuo lavoro

08

Branch: lavorare in parallelo

Un branch (ramo) è una linea di sviluppo indipendente. Tecnicamente è solo un puntatore mobile a un commit: leggerissimo da creare. Servono a sviluppare una funzionalità senza disturbare il ramo principale (main), per poi riunire il lavoro con un merge.

Un puntatore speciale, HEAD, indica il ramo su cui stai lavorando ora.

Creare, spostarsi, unire

git branch                     # elenca i rami (* = quello attuale)
git switch -c feature          # crea "feature" e ci si sposta
# ...lavori e fai dei commit sul ramo feature...

git switch main                # torna sul ramo principale
git merge feature              # unisce il lavoro di feature in main

git branch -d feature          # cancella il ramo, ormai unito
switch e restore sono i nuovi nomi

Git ha diviso il vecchio, sovraccarico git checkout in due comandi più chiari: git switch per cambiare ramo e git restore per ripristinare file. Vedrai ancora checkout in molte guide: fa entrambe le cose.

Condividere un branch

git push -u origin feature     # pubblica il ramo sul remoto

09

Tag: marcare le versioni

Un tag è un'etichetta fissa su un commit, usata per segnare punti importanti, tipicamente i rilasci (v1.0.0). A differenza di un branch, un tag non si muove: resta inchiodato a quel commit per sempre.

Esistono due tipi di tag. Quello annotato (con -a) è da preferire per i rilasci: salva autore, data e un messaggio.

git tag                              # elenca i tag esistenti

git tag v1.0.0                       # tag "leggero": solo un nome
git tag -a v1.0.0 -m "Primo rilascio pubblico"   # tag annotato

git show v1.0.0                      # dettagli del tag e del commit

Importante: git push da solo non invia i tag. Vanno inviati esplicitamente.

git push origin v1.0.0               # un tag specifico
git push origin --tags               # tutti i tag in una volta

10

Intermezzo: e se ti perdi?

Capita a tutti, all'inizio, di sentirsi così. C'è una celebre vignetta di xkcd che fotografa perfettamente il primo impatto con Git, e la verità è che ci passano davvero tutti.

La buona notizia: non è vero che “nessuno sa come si usa”. Con i pochi concetti di questa guida, le tre aree, i commit, i rami, hai già la mappa per non doverti mai più cancellare il progetto e riscaricarlo.
